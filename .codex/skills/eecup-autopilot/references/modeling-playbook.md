# 建模手册

## 模型阶梯

始终按顺序推进：

1. 题意与数据理解：明确输入、输出、约束、指标。
2. 调研与路线选择：联网查公开资料，形成候选方案，暂停让用户选择。
3. 基准模型：简单、透明、快速。
4. 主模型：更强，但仍能解释。
5. 工程修正：加入约束、物理意义或运行场景。
6. 验证分析：比较、压力测试和解释。

所选模型必须完全贴合赛题场景，不能生搬硬套；在贴合的基础上可以做一些创新
不要跳过基准模型。基准是向评委证明改进模型有效的证据，也是后续多模型比较的统一评价底座。

主模型也不能未经选择直接实现。先写 `00_problem/model_options.md`。
## 基准先行

基准模型不占用候选路线名额。至少 3 个候选路线在调研选型关卡确定；基准模型在用户确认路线执行方式后，用来固定数据版本、评价指标、随机种子和验证切分。

进入多模型隔离代码顺序试验前，必须先完成并提交：

- 可运行的基准模型；
- 统一数据版本和预处理口径；
- 统一评价指标；
- 固定随机种子；
- 统一验证切分或场景集合。

## 多模型隔离代码顺序试验

多模型顺序试验只发生在模型路线关卡之后。候选路线必须先写入 `00_problem/model_options.md`，并等待用户确认路线执行方式：单路线推进，或多路线顺序隔离比较。

确认后按以下规则执行：

1. 若选择多路线顺序隔离比较，先完成并提交统一基准模型和评价脚手架。
2. `00_problem/model_options.md` 必须列出被选入比较的路线、试验顺序和最终选择前的完成条件。
3. 不为每条候选路线默认创建 Git 分支；同一主线下按顺序逐个实现候选模型。
4. 公共代码放在 `02_src/common/`、`02_src/baseline/` 和 `02_src/evaluation/`；候选模型代码只写入 `02_src/models/<model_id>/`。
5. 每次只推进一个候选模型。该模型完成或失败后，保存标准结果包、本地提交，再进入下一条已确认路线。
6. 候选模型结果保存到 `03_results/models/<model_id>/`，至少包含 `metrics.csv`、`summary.md` 和必要图表目录 `figures/`。
7. 所有候选模型使用同一基准模型确定的数据版本、评价指标、随机种子和验证切分；若必须不同，写入 `06_management/experiment_log.csv`。
8. 比较阶段从 `03_results/models/*/metrics.csv` 汇总统一比较表，优先保存为 `03_results/tables/model_comparison.csv`。
9. 最终选择前更新 `00_problem/model_options.md`、`06_management/decision_log.md` 和 `04_paper/paper_draft.md`，列出各路线优劣、指标差异、论文表达价值和推荐结论，然后暂停等待用户确认。

子 agent 可以负责单个候选模型，但必须遵守相同的隔离目录和标准结果包约束；最终汇总和推荐由主流程完成。

## 比较完成门槛

进入最终模型选择前，所有被选入比较的候选路线都必须满足以下任一条件：

- 已完成同一口径的可复现实验，并写入统一比较表；
- 因不可行、效果差、运行成本高或不可解释而失败，并在 `MODEL_ISSUE_REPORT.md` 或统一比较表中记录失败原因、已尝试内容和是否放弃。

不得只展示成功模型。任一候选路线出现不优于基准、约束违反、结果不稳定或不可解释时，仍按 `references/model-recovery.md` 判断是否写异常报告并暂停。若单个模型失败不影响统一数据、指标和约束口径，可记录失败并继续下一条已获用户确认的候选路线；若暴露系统性问题，则写 `06_management/MODEL_ISSUE_REPORT.md` 并暂停。

## 建模图表产出

建模代码阶段不得只输出指标表。每个小问原则上至少准备数据理解类图、模型或算法流程图、结果对比类图；若涉及敏感性、鲁棒性或误差分析，还应准备验证分析类图。

论文第 `2. 问题分析` 章节应规划 1 张问题分析总体框架图，用于固定全篇“题意拆解 -> 数据输入 -> 模型构建 -> 求解验证 -> 结果输出”的叙事路线。每个具体模型建立与求解章节不再重复放问题分析框架图，只保留算法流程图、模型结构图、求解流程图或结果图。

数据处理阶段优先产出：

- 缺失值或异常值概览图；
- 关键变量分布图；
- 时间序列、负荷曲线、电价曲线、功率曲线或场景曲线；
- 必要时补充相关性热力图或特征关系图。

基准模型阶段优先产出：

- 基准预测、优化或评价结果图；
- 基准误差、残差或约束违反情况图；
- 基准模型与题面关键约束的对应示意。

主模型阶段优先产出：

- 主模型算法流程图；
- 主模型与基准模型对比图；
- 关键变量、调度结果、优化路径、排名结果或分类结果图。

验证分析阶段优先产出：

- 灵敏度曲线、箱线图、误差分布图、残差图或鲁棒性场景对比图；
- 优化类问题优先展示约束满足情况、成本或收益变化、资源使用曲线；
- 预测类问题优先展示真实值与预测值对比、残差分布、极端场景误差；
- 评价类问题优先展示权重敏感性、排名稳定性、指标贡献。

图表输出保存到 `03_results/figures/` 或 `03_results/models/<model_id>/figures/`。论文采用的图必须在 `04_paper/paper_draft.md` 中登记插入位置、来源脚本、来源数据、输出路径和支撑结论。

## A 题常见模式

| 类型 | 基准模型 | 主模型 | 验证方式 |
|---|---|---|---|
| 预测 | 均值、线性回归、ARIMA、随机森林 | XGBoost、LightGBM、LSTM、Transformer，前提是有理由 | 顺序切分、MAE/RMSE/MAPE、残差、极端样本 |
| 优化 | 贪心或线性松弛 | MILP、动态规划、遗传算法、粒子群、滚动优化 | 可行性、场景对比、灵敏度 |
| 评价 | 加权求和 | 熵权法、AHP、TOPSIS、PCA、模糊评价 | 排名稳定性和权重敏感性 |
| 调度 | 最早优先启发式 | MILP、CP-SAT、元启发式 | 约束违反、成本对比 |
| 能源系统 | 规则调度 | 含功率平衡约束的优化 | SOC、容量、负荷、天气、电价场景 |

## 电气工程纪律

检查模型中是否应包含：

- 功率平衡
- 电压、电流、容量限制
- 爬坡限制
- 储能荷电状态
- 效率和损耗
- 安全边界
- 可靠性或备用裕度
- 分时电价
- 天气、负荷或场景不确定性

如果省略，需要说明原因。

## 最低验证要求

每个重要结果都要：

- 记录数据来源和脚本；
- 报告带单位的指标；
- 与基准模型比较；
- 包含至少一项压力测试或灵敏度检查；
- 说明工程意义，而不只是给出数学数值。

## 效果不好时的判断

以下情况视为模型效果异常：

- 相比基准没有提升，或提升过小；
- 优化不可行或约束违反过多；
- 预测在关键场景下失真；
- 排名或评价结论对权重极敏感；
- 模型运行时间不可接受；
- 结果无法解释到题目工程背景。

触发后读取 `references/model-recovery.md`，写 `06_management/MODEL_ISSUE_REPORT.md`，暂停等待用户选择。

## 结果卫生

固定随机种子。指标保存为 CSV。尽量从 CSV 生成论文表格。不要手工编造论文数字。
