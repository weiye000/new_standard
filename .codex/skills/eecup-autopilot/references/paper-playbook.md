# 论文手册

## 冲奖论文形态

评委应能快速看出：

1. 题意理解准确。
2. 模型适合工程背景。
3. 结果可复现且有具体数值。
4. 改进方法优于基准模型。
5. 局限性诚实且边界清楚。

## 默认章节结构

除非官方模板另有要求，使用以下结构：

1. 摘要
2. 问题重述
3. 模型假设与符号说明
4. 数据处理与描述性分析
5. 问题一模型、求解与结果
6. 问题二模型、求解与结果
7. 问题三模型、求解与结果
8. 灵敏度、鲁棒性与误差分析
9. 模型评价与改进
10. 参考文献
11. 附录和支撑材料说明

## 模板处理规则

比赛开始后先检查官方题包、官网通知和论文规范是否提供 Word、LaTeX 或其他格式模板。

- 有官方模板：最终稿必须使用官方模板；`paper_draft.md` 只作为内容母稿。
- 无官方模板：按电工杯论文规范自行排版，默认优先生成 PDF。
- 第三方 LaTeX 模板：只可作为排版参考，不能写成“官方模板”。
- 任何模板选择都要记录在 `PHASE_REVIEW.md`。

电工杯论文规范: https://shumo.neepu.edu.cn/jszz/lwgf.htm
必须严格遵守

## Markdown 同步初稿

从题意拆解开始维护：

```text
04_paper/paper_draft.md
```

它不是最后定稿，而是持续积累的论文骨架。每个阶段至少更新一次：

| 阶段 | 更新内容 |
|---|---|
| 题意拆解 | 问题重述、任务拆解、符号草案 |
| 调研选型 | 方法综述、模型选择理由、候选方案摘要 |
| 基准模型 | 数据处理、基准模型、初步结果表 |
| 主模型 | 模型公式、算法流程、对比结果 |
| 验证分析 | 灵敏度、鲁棒性、误差和局限性 |
| 最终整理 | 摘要、结论、支撑材料说明 |

数字结论必须标注来源路径，例如 `03_results/tables/main_metrics.csv`。未生成的数字写“待生成”，不要先编。

## 摘要写法

每个任务用一个紧凑段落交代：

- 任务目标；
- 采用方法；
- 关键定量结果；
- 验证或对比；
- 工程解释。

避免没有数值支撑的“效果较好”等空话。

## 获奖论文写作套路

获奖论文通常不是堆模型，而是让评委快速看到“题意准、模型合适、结果可信、表达清楚”。

### 摘要

把摘要写成微型技术报告：

```text
针对问题 X，本文先……；然后建立……模型；得到……关键结果；
与……基准相比……；灵敏度/鲁棒性分析表明……；该结论说明……
```

摘要必须包含具体方法、关键数字、验证方式和工程解释。没有生成的数字先写“待生成”，不要写空泛评价。

### 正文小问结构

每个小问固定使用：

```text
问题分析 -> 模型建立 -> 求解过程 -> 结果解释 -> 检验/敏感性 -> 小结
```

每个小问结尾用 2-4 句话回答：本小问要解决什么、模型得出什么、结果为什么可信、对后续小问有什么作用。

### 模型表达

每个模型必须回答：

- 为什么这个模型适合本题；
- 变量、目标、约束和评价指标分别对应题目中的什么；
- 相比基准模型改进在哪里；
- 哪些工程约束或实际含义被保留下来。

避免只堆公式。公式后必须有变量解释和建模意图。

### 图表表达

图表必须服务结论：

- 图前说明为什么画这张图；
- 图题和坐标轴写清单位；
- 图后解释发现，不只说“如图所示”；
- 表格优先展示能支撑结论的关键指标。

### 结果段落

结果段落至少包含：

- 数字；
- 单位；
- 对比对象；
- 指标变化；
- 工程含义；
- 结果来源路径。

### 模型评价

结尾评价模型时写具体优缺点，不写“模型较好、仍有不足”等空话。优点对应解决了什么问题，缺点对应什么场景下可能失效。

## 评委检查清单

最终整理前确认：

- 摘要包含具体数字。
- 每个小问都有直接答案。
- 符号先定义后使用。
- 表格和坐标轴包含单位。
- 图在灰度下仍可读。
- 论文不依赖隐藏聊天上下文。
- 结论与生成的结果文件一致。
- 支撑材料能复现关键图表。
- `paper_draft.md` 已覆盖所有小问，不依赖聊天历史。

## 文风规则

优先使用短技术句。详细解释为什么选择每个模型。除非必要，不在正文放代码。长推导、参数表和额外图放入附录或支撑材料，不使用过分晦涩难懂的句子或者表达。
