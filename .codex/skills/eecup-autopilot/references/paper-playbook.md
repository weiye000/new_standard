# 论文手册

## 冲奖论文形态

评委应能快速看出：

1. 题意理解准确。
2. 模型适合工程背景。
3. 结果可复现且有具体数值。
4. 改进方法优于基准模型。
5. 局限性诚实且边界清楚。

## 默认章节结构

使用以下结构：

摘要
1. 问题重述
    1.1 问题背景
    1.2 问题提出
2. 问题分析
3. 模型假设
4. 符号说明
5. 数据处理与描述性分析
6. 模型建立与求解
    6.1 问题1 模型建立与求解
        6.1.1 问题1求解思路
        6.1.2 问题1模型建立
        6.1.3 问题1求解与分析
    6.2 问题2 模型建立与求解
        6.2.1 问题2求解思路
        6.2.2 问题2模型建立
        6.2.3 问题2求解与分析
    6.3 问题3 模型建立与求解
        6.3.1 问题3求解思路
        6.3.2 问题3模型建立
        6.3.3 问题3求解与分析
9. 模型分析检验（灵敏度、鲁棒性与误差分析等等）
10. 模型评价与改进
11. 参考文献
12. 附录和支撑材料说明

## 模板处理规则

比赛开始后先检查是否提供 Word、LaTeX 或其他格式模板。

- 有官方模板：最终稿必须使用官方模板；`paper_draft.md` 只作为内容母稿。
- 无官方模板：按下面论文排版规则用Latex自行排版，默认优先生成 PDF。
- 任何模板选择都要记录在 `PHASE_REVIEW.md`。

## 论文排版规则
- 论文面页设置为A4纸，上下左右各留出2.5厘米的页边距，正文字体为宋体小四号字。

- 论文第一页：封面页，填写报名序号及论文题目（题目不用自拟），报名序号填写错误后果自负。

- 论文第二页：论文题目、摘要和关键词（摘要、关键词无需译成英文），并从此页开始编写页码，页码必须位于每页页脚中部，用阿拉伯数字从“1”开始连续编号。注意：摘要内容简明扼要，篇幅不超过一页。

- 从第三页开始是论文正文（不要目录，控制在25页以内）；正文之后是论文附录（页数不限）。

- 引用别人的成果或其他公开的资料(包括网上查到的资料) 必须按照规定的参考文献的表述方式在正文中引用和参考文献中明确列出。正文引用处用方括号标示参考文献的编号，如[1][3]等；引用书籍还必须指出页码。参考文献按正文中的引用次序列出，其中书籍的表述方式为：

[编号] 作者，书名，出版地：出版社，出版年，页码.

参考文献中期刊杂志论文的表述方式为：

[编号] 作者，论文名，杂志名，卷期号：起止页码，出版年.

参考文献中网上资源的表述方式为：

[编号] 作者，资源标题，网址，访问时间（年月日）.

## Markdown 同步初稿

从题意拆解开始维护：

```text
04_paper/paper_draft.md
```

它不是最后定稿，而是持续积累的论文骨架。每个阶段至少更新一次：

| 阶段 | 更新内容 |
|---|---|
| 题意拆解 | 问题重述、任务拆解、符号草案 |
| 调研选型 | 方法综述、模型选择理由、候选方案摘要 |
| 基准模型 | 数据处理、基准模型、初步结果表 |
| 主模型 | 模型公式、算法流程、对比结果 |
| 验证分析 | 灵敏度、鲁棒性、误差和局限性 |
| 最终整理 | 摘要、结论、支撑材料说明 |

数字结论必须标注来源路径，例如 `03_results/tables/main_metrics.csv`。未生成的数字写“待生成”，不要先编。

## 摘要写法

每个任务用一个紧凑段落交代：

- 任务目标；
- 采用方法；
- 关键定量结果；
- 验证或对比；
- 工程解释。

避免没有数值支撑的“效果较好”等空话。

### 摘要

把摘要写成微型技术报告：

```text
针对问题 X，本文先……；然后建立……模型；得到……关键结果；
与……基准相比……；灵敏度/鲁棒性分析表明……；该结论说明……
```

摘要必须包含具体方法、关键数字、验证方式和工程解释。没有生成的数字先写“待生成”，不要写空泛评价。

### 正文小问结构

每个小问固定使用：

```text
问题分析 -> 模型建立 -> 求解过程 -> 结果解释 -> 检验/敏感性 -> 小结
```

每个小问结尾用 2-4 句话回答：本小问要解决什么、模型得出什么、结果为什么可信、对后续小问有什么作用。

### 模型表达

每个模型必须回答：

- 为什么这个模型适合本题；
- 变量、目标、约束和评价指标分别对应题目中的什么；
- 相比基准模型改进在哪里；
- 哪些工程约束或实际含义被保留下来。

避免只堆公式。公式后必须有变量解释和建模意图。

### 图表表达

图表必须服务结论：

- 图前说明为什么画这张图；
- 图题和坐标轴写清单位；
- 图后解释发现，不只说“如图所示”；
- 表格优先展示能支撑结论的关键指标。

### 图表美学

论文图表应使用统一、克制、专业的视觉风格，避免默认粗糙配色、彩虹色、过密网格和无意义 3D 效果。

- 推荐使用色盲友好、印刷友好的配色，例如蓝、橙、绿、灰等低饱和组合；同一类指标在全文中保持同一颜色语义。
- 图表必须包含清晰坐标轴、单位、图例、必要注释和可读字号；坐标轴范围不能误导结论。
- 论文最终图优先导出为 `PDF` 或 `SVG` 等矢量图，同时保留 `PNG` 预览；位图分辨率建议不低于 300 dpi。
- 图表标题、图注和正文解释要形成闭环：图前说明目的，图中展示证据，图后解释发现和工程意义。
- 不为“显得丰富”而堆图；正文放关键图，补充图放附录或支撑材料。

### 流程图插入位置

流程图用于帮助评委快速理解技术路线、数据处理、模型结构和求解步骤，不用于替代结果证据。

- `2. 问题分析` 中固定插入 1 张问题分析总体框架图，展示全篇各小问的分析路线和逻辑关系。
- `数据处理与描述性分析` 中可插入数据处理流程图，展示原始数据、清洗、特征构造、数据切分、评价数据集和输出文件。
- `模型建立与求解` 章节中只插入算法流程图、模型结构图、求解流程图或结果图，不插入问题分析框架图。
- `验证分析` 中只插入验证、误差、鲁棒性或灵敏度相关图。
- 多模型比较时可插入模型比较流程图，展示统一数据版本、统一指标、基准模型、候选模型、结果包和最终选择逻辑。
- 模型异常或修正后，若论文中需要解释方法调整，可以插入模型修正流程图；不要强制每次异常都画。
- 支撑材料中保留更完整的代码运行流程图或复现流程图，正文只放评委最需要看的关键流程。

### 问题分析总体框架图

绘制前先读取 `references/problem-analysis-framework-chart.md`。

问题分析总体框架图只放在 `2. 问题分析` 章节，用于展示全篇各小问的分析路线和逻辑关系，不在每个具体模型建立与求解章节中重复插入。

该图可以采用多分区或多子图形式，每个分区对应一个小问。每个小问分区应包含：核心目标、输入数据、关键变量或约束、预处理步骤、拟采用模型、验证方式和输出结论。

模型建立与求解章节中不再插入问题分析框架图；如需插图，只放算法流程图、模型结构图、求解流程图或结果图。问题分析框架图负责“为什么这样做”，模型求解图负责“具体怎么做”。

图后用一小段文字解释各小问之间的逻辑关系，例如问题一提供基础关系分析，问题二在此基础上优化策略，问题三进一步进行动态或个性化扩展。图中不得包含未选择、未执行或未验证的模型步骤。

### 图表工具与技能

正式建模结果图、流程图和算法图必须可复现、可追溯、可修改，优先由代码生成。

- 数据图和结果图优先使用 `Python matplotlib + seaborn`，或项目实际使用的 MATLAB 绘图。
- 复杂交互探索可以临时使用 `plotly`，最终论文仍导出静态图。
- 流程图优先使用 `Mermaid`、`Graphviz`、`TikZ` 或代码绘制的矢量流程图。
- 最终 PDF 检查可使用 `pdf` skill 做页面渲染和版面检查。
- `imagegen` 只在需要非数据型示意背景、封面视觉或透明装饰素材时考虑；不得用 AI 生成图替代正式建模结果图、流程图或算法图。
- 如果最终论文使用 LaTeX，应在写作阶段明确把流程图和结果图插入对应章节，并保留 `\includegraphics` 的源路径注释；若使用 Word，也要保留图源文件和生成脚本路径。

### 结果段落

结果段落至少包含：

- 数字；
- 单位；
- 对比对象；
- 指标变化；
- 工程含义；
- 结果来源路径。

### 模型评价

结尾评价模型时写具体优缺点，不写“模型较好、仍有不足”等空话。优点对应解决了什么问题，缺点对应什么场景下可能失效。


## 文风规则

除非必要，不在正文放代码。长推导、参数表和额外图放入附录或支撑材料，不使用过分晦涩难懂的句子或者表达，语言流畅专业，排版整洁优美大方。
