# 问题分析总体框架图参考

## 定位

问题分析总体框架图只放在论文 `2. 问题分析` 章节，用于让评委快速看懂全篇各小问的分析路线和逻辑关系。它不是算法流程图，也不是结果图；它负责回答“为什么这样拆题、每个小问沿什么路线推进、各小问之间如何衔接”。

每个具体模型建立与求解章节不再重复插入问题分析框架图。模型章节如需插图，只放算法流程图、模型结构图、求解流程图、结果图或验证图。

## 推荐图形形态

优先采用多面板总览图。若题目包含多个小问，可以按 2x2、1x3 或 2x3 布局组织，每个面板对应一个小问。每个面板内部使用浅色背景、分区边框、短文本模块和少量主箭头。

面板标题格式：

```text
问题一：核心任务名称
问题二：核心任务名称
问题三：核心任务名称
```

每个面板建议包含以下信息：

- 核心目标：本小问最终要解决什么；
- 输入数据：题面数据、附件数据、外部可用信息或上一问输出；
- 关键变量或约束：核心指标、变量、工程约束、评价指标；
- 预处理或特征构造：清洗、标准化、编码、分组、时间序列处理、特征组合；
- 模型方法：基准模型、主模型或策略模型的简短名称；
- 验证方式：统计检验、误差分析、鲁棒性、敏感性、约束检查；
- 输出结论：本小问要交付的表、图、策略、预测、评价或优化结果。

## 版式参考

可以参考以下结构组织单个小问面板：

```text
┌──────────────────────────────┐
│ 问题一：核心任务标题          │
├──────────┬───────────────────┤
│ 数据输入 │ 数据1 / 数据2 / 字段 │
│ 数据处理 │ 缺失处理 / 标准化 / 特征构造 │
├──────────┼───────────────────┤
│ 模型构建 │ 变量 -> 模型方法 -> 约束或目标 │
│ 求解验证 │ 检验方法 / 对比对象 / 鲁棒性 │
├──────────┴───────────────────┤
│ 输出：关键结论、策略或结果表   │
└──────────────────────────────┘
```

如果小问逻辑更接近优化或决策，可以使用纵向链条：

```text
目标函数或评价目标
      ↓
关键变量与约束
      ↓
模型或算法
      ↓
策略优化或求解
      ↓
最终输出
```

如果小问逻辑更接近预测或分类，可以使用特征到模型的结构：

```text
静态特征 + 时序特征
      ↓
特征融合或筛选
      ↓
预测 / 分类 / 评价模型
      ↓
误差、稳定性或可解释性检验
      ↓
预测结果或诊断结论
```

## 视觉规范

图形风格应统一、清晰、克制，避免复杂装饰。

- 每个小问面板使用一种低饱和主色，例如橙、蓝、绿、红；同一小问的标题边框、阶段标签和强调线保持同色系。
- 大区块使用浅色背景，模块框使用白底或极浅色底；阶段分区可用虚线边框。
- 箭头只连接主要逻辑路径，避免把所有模块都强行连线。
- 文本尽量短，模块名优先使用 4 到 10 个字；复杂解释放到图后正文。
- 字体、字号、边框粗细、圆角、间距在全图保持一致。
- 论文最终图优先导出为 `PDF` 或 `SVG`，同时保留 `PNG` 预览。

## 内容边界

图中只能出现已经由题意拆解、模型选择或后续实验支撑的内容。

- 不写未选择的主模型作为最终路线；
- 不写未运行的实验结果；
- 不把“待生成”的数字画成确定结论；
- 不为追求对称而添加实际不会执行的步骤；
- 不把算法细节全部塞进问题分析图，细节留给模型章节的算法流程图。

## 论文写法

在 `2. 问题分析` 中插入该图后，图前用 1 到 2 句话说明绘图目的，图后用一段话说明小问之间的关系。

推荐表达：

```text
为明确各小问之间的递进关系，图 X 给出了本文的问题分析总体框架。总体上，问题一用于……，问题二在问题一结果基础上……，问题三进一步……，最终形成……。
```

`paper_draft.md` 中必须登记：

- 图名；
- 插入位置：`2. 问题分析`；
- 覆盖小问；
- 来源文件或脚本；
- 输出路径；
- 支撑作用；
- 当前状态。

## 制作方式

初稿阶段可以先用 Mermaid 或纯文本草图梳理结构；定稿阶段优先用 Python、TikZ、Graphviz 或绘图软件生成可修改的矢量图。正式论文图不得使用 AI 生成图替代，因为该图必须可追溯、可修改，并能与正文、代码和结果文件对应。

若使用代码绘制，建议保存源文件到 `02_src/`，输出保存到 `03_results/figures/`；若使用手工绘图工具，也应保存可编辑源文件，并在论文草稿中登记源文件路径。

## 完成检查

插入论文前确认：

- 图只出现在 `2. 问题分析`；
- 每个核心小问都有一个清晰分区；
- 每个分区都包含目标、输入、方法、验证和输出；
- 图中模块能对应正文、数据、代码或结果文件；
- 图后文字解释了各小问之间的逻辑关系；
- 模型章节没有重复放问题分析框架图。
